###################################
#Introduction                     #
###################################
- Communication - Information Exchange
- Two Parties can communicate if they agree upon same set of rules / contract / agreement
- Interface : In Computing, a Interface is a shared boundary between two entities that allows information exchange over specific entry points in a specified manner 
- API (Application Programming Interface) - Interface (Set of Rules/Contract) between two Applications to communicate in a defined manner
- Web API : Refers to API used by web applications to communicate over internet using web based protocol http
- Postman is a Tool used for Designing and Testing APIs
- Postman App comes in two flavours : Desktop & Web
- Documentation is important to understand and use an API
- Every API has a base URL (main address) & Endpoints (specific relative address appended to base url) for every functionality and data/resource
- Postman Web app has same UI except extra option for selecting a agent (Cloud : Sends requests from Postman Server[Limited Attempts], Browser [Limited Capability] , Desktop Agent)
- HTTP : Hyper Text Transfer Protocol | HTTPS : HTTP Secured that encrypts data being exchanged
- Postman (Client) interacts with Server via API 
- Postman sends a Request (URL,METHOD,HEADERS,BODY) and Receives a Response (STATUS CODE,HEADERS,BODY)


#################################################
#API Basics & Postman Functionalities           #
#################################################
- Collections : Folders in Workspace to organize and group Requests specific to an API or Use Cases of an API
  *Can be shared with other collaborators via multiple options
  *Folders : sub-Folders can be created inside Collections to further group/reuse requests (Happy Path , Negative Scenario)
- Variables : Used to store configurations to make requests flexible/efficient. Scope of Variables : {Collection, Environment, Global}
  * To see variables of a particular collection tap on the collection under workspaces
  * We can share/un-share value of variables in a collection (useful in Team Collaboration/Monitors/Scheduled Runs)
  * Variables can be referred inside JSON in body by using syntax : {{x}}
  * Random Variables : "{{$XYZ....}}" feature provided by Postman to insert random values into our JSON to test our APIs

- Responses received from API in body are displayed in "Pretty" form by Default to make it easy to understand. We have options to see "Raw" form. Also "Preview" mode tries to display the data in best possible form.

#API Authentication: Unlike Websites which uses Username Password to provide access, APIs use access tokens to grant access to the resource.
 .Bearer Token one of the standard ways , is set as parameter in Header (Its a Hidden Parameter in Header Section of the request)
 .There are other standard authentications mechanisms used by APIs

#Parameters (Ways to Exchange information):
1. Query Parameters : Optional/Mandatory Key-value pairs appended after ? to a endpoint to modify the response
  *API doesn't respond(ignore) to a query parameter not defined in documentation
  *We can also directly specify query parameters in the URL stringed together with "&" character which acts as delimiter

2. Path Variables : Specifies named values which are translated to form dynamic segment of the URL path while making the request. Specified with ":" symbol in API
  *Post Substitution translates into a proper API URL with Endpoint
  *They are always mandatory unlike Query Parameters

3. Body : Content (generally JSON but may be any other type of data) we sent in a request (optional) and receive as a response  [If it is not mandatory it can be skipped]

4. Headers: Key Value Pairs used to specify meta data for Request or in a response. 
  *Postman Automatically generates many headers.
  *We can add header under headers section and override existing if any.


* Bulk Edit Feature of Postman under Request Section helps to copy Key-Value Pairs easily

#########################################################
#HTTP Request Methods/Verbs(General API Design Pattern):#
#########################################################
1. GET : Used to retrieve Data. It doesn't require body and parameters are in headers and URL

2. POST : Used to create new Data. Generally has a request body and parameters are in body , headers & URL 

3. PATCH : Used to update one or two properties of an existing data object.

4. PUT : Used to replace an existing data object completely.

5. DELETE : Used to delete and existing data object

6. HEAD : Used to determine if Endpoint is valid/Up. (Lightweight & no body )



**Note: 
1.All Endpoints may not support all methods and you may receive status code 404 not found
2.Always Consult APIs to be sure which method to use for CRUD operations

#Automation Testing with Postman  #
###################################
- Postman Allows to write Scripts "JavaScript Snippets" to add dynamic behaviour to Workflows (Setting Variables for chaining ,testing)
- In Request Section , Under Scripts Tab we have Option to add "Pre-Request" & "Post-Response" Scripts
- We Can utilize snippets from snippets library available in Postman
- In Response Section, Under Test Results Tab we can view the Test Results
- JS Variables and Postman Variables are different things
- Postman Variables can be used in Scripts and in request builder section tabs with syntax {{x}}
- In Postman we use callback functions as scripts are to be executed once response is received
- For Callback functions in postman use anonymous than arrow
- In API Testing we send and receive data in form of JSON. This data needs to be converted to/from JS Objects for Testing
- We can add Tests at Collection Level and Folder level to reuse them. All Test of Parent Level will be executed by default
  so duplication of code is not required.
- We Can used Doscumentation Option to document about test under Docs tab and about folders/collections under overview tab
- Postman Variables helps to implement request chaining 
 
- pm.test(testName,specFunction) => to specify test assertions
  testName : Name of Test (string) Ex : "Status code is 200"
  specFunction : Callback function with test assertions. Ex : () => pm.response.to.have.status(200)

- pm.response.text() : Return response as a text
- pm.response.json() : Returns response as a JS object if response is a json
- JSON.parse(pm.response.text()) => parses text to a JS Object
- pm.expect(x) => Assert something about x. used with a BDD Assertion Library chai.js
  .to.be.equal , to.include, to.have.property , to.haveOwnProperty , to.be.true to.be.a e.t.c

- pm.collectionVariables.set(key,value) =>Creates or updates Collection Variable (key : String , Value : Any Value)
- pm.collectionVariables.get(key) =>reads Collection Variable (key : String )
- pm.collectionVariables.unset(key) =>Deletes Collection Variable (key : String )
- pm.collectionVariables.clear() =>Deletes All Collection Variables

- When adding data to variable via script , Data Type is preserved. In Manual addition same is not preserved.

###################################
#Testing Tips                     #
###################################
1.Test with All Possible Permutation of Query Parameter Values (Numbers, Alphabets, Alphanumeric with special characters, boundary values)
  Discuss abnormal findings with developers
2.For Performance issues, APIs limits responses to certain specified limit by breaking data response up into pages (Known as pagination)
3.API are build around complex Business Rules. Same needs to be understood properly to design and test APIs (Think for all end user flows)
4.Read API Documentation carefully to understand API its usage and design Test Cases accordingly
5.Best way to navigate through API is via Use Cases
6.API generally creates unique ID for data objects to manage and avoid conflicts
7.Always Hide your API Keys and Make sure if your Collection Visibility is Public then Keys Information should be hidden 
8.Fucnctional Testing : Testing whether API gives correct response for a request as per requirements/bussiness logic.
  a. Status Code Check : Write Tests to Verify Status Codes (Write Assertions for status codes)
  b. Response Body : Write Assertions for Response Body (Value,Type,Range)


9.Happy Path Testing : Testing main business use case works (Testing Most Important Scenarios works)
  Negative Testing : Testing how API behaves in exceptional (Testing for scenarios where API should Fail)
  Boundary Testing : Testing API for boundary/invalid inputs

10.TDD : Test Driven Development
  . Business Requirements are First Translated into Tests and then Code is written to pass the tests.
  . For new Requirement first tests are written, If Tests fail then only code is altered
